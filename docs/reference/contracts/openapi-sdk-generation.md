# Python service → TypeScript SDK generation (OpenAPI)

How the `nogai` Next.js app gets typed TypeScript clients for the Python
FastAPI "mini" services. Contract direction: **Python owns the contract**
(FastAPI emits OpenAPI), TypeScript consumes it.

## Source of truth (observed)

Each backend service is a FastAPI app that auto-emits an OpenAPI document at
`/openapi.json`. The apps declare `title` / `version` on the `FastAPI(...)`
constructor (e.g. `backend/mini_document_search/main.py`:
`title="Document Search Service"`, `version="0.1.0"`). There is **no exported
spec file** committed — generation reads the spec from a *live, locally
running* service.

Four services participate, each on a fixed port (from `compose.yaml` and each
service's `uvicorn.run(..., port=...)`):

| Service              | Port | SDK config                                    | SDK output dir                  |
|----------------------|------|-----------------------------------------------|---------------------------------|
| mini_knowledge       | 8007 | `conf/openapi-ts/mini_knowledge.config.ts`    | `src/sdk/mini_knowledge`        |
| mini_workflow        | 8008 | `conf/openapi-ts/mini_workflow.config.ts`     | `src/sdk/mini_workflow`         |
| mini_document_search | 8009 | `conf/openapi-ts/mini_document_search.config.ts` | `src/sdk/mini_document_search` |
| mini_preview         | 8011 | `conf/openapi-ts/mini_preview.config.ts`      | `src/sdk/mini_preview`          |

## Generator

`@hey-api/openapi-ts` (pinned `0.82.3` in `frontend/nogai/package.json`
devDeps). One config file per service, all identical in shape. Full example
(`conf/openapi-ts/mini_knowledge.config.ts`):

```ts
import { defineConfig } from "@hey-api/openapi-ts";

export default defineConfig({
  input: "http://localhost:8007/openapi.json",
  output: "src/sdk/mini_knowledge",
  plugins: ["@hey-api/client-fetch"],
});
```

Key facts:
- `input` is a **live URL** — the target service must be running locally on the
  matching port before regeneration.
- `plugins: ["@hey-api/client-fetch"]` selects the fetch-based client runtime.

## npm scripts (observed)

From `frontend/nogai/package.json`:

```json
"gen:mini_workflow": "openapi-ts -f conf/openapi-ts/mini_workflow.config.ts && tsx scripts/patch-sse-streaming.ts src/sdk/mini_workflow",
"gen:mini_knowledge": "openapi-ts -f conf/openapi-ts/mini_knowledge.config.ts && tsx scripts/patch-sse-streaming.ts src/sdk/mini_knowledge",
"gen:mini_document_search": "openapi-ts -f conf/openapi-ts/mini_document_search.config.ts && tsx scripts/patch-sse-streaming.ts src/sdk/mini_document_search",
"gen:mini_preview": "openapi-ts -f conf/openapi-ts/mini_preview.config.ts"
```

Each script: (1) runs `openapi-ts` against the config, then (2) for the three
SSE-capable services, runs a post-generation patch (see below). `mini_preview`
has no streaming endpoints so it skips the patch.

## Post-generation patch (observed)

`scripts/patch-sse-streaming.ts` rewrites `core/serverSentEvents.gen.ts` in the
generated SDK. It replaces the generated `TextDecoderStream` pipe with a manual
`TextDecoder`, working around a Next.js production runtime bug
(`controller[kState].transformAlgorithm is not a function`). The script is
idempotent (bails if `const decoder = new TextDecoder();` is already present)
and pattern-guarded (warns if the generated structure changed and the regex no
longer matches). This is the one place hand-maintained logic reaches into
generated output; it re-applies deterministically on every regen.

## Generated SDK layout (observed)

Per service dir under `src/sdk/<service>/`:
- `types.gen.ts` — request/response TypeScript types
- `sdk.gen.ts` — one exported function per operation (named after the operation
  id, e.g. `indexDocumentIndexPost`, `searchDocumentsSearchPost`)
- `client.gen.ts` — a default `client` pre-configured with the dev `baseUrl`
  (e.g. `baseUrl: 'http://localhost:8007'`)
- `client/`, `core/` — the client-fetch runtime
- `index.ts` — `export * from './types.gen'; export * from './sdk.gen';`

All files carry `// This file is auto-generated by @hey-api/openapi-ts`.

## Consumption (observed)

Two consumption patterns:

1. **Direct operation import** in Next.js routes / agent tools / pages, e.g.
   `import { previewFromUploadV1PreviewUploadPost } from "@/sdk/mini_preview";`
   and `import { LegislationManifestation } from "@/sdk/mini_knowledge";`
   (`@/sdk/...` resolves to `src/sdk/...`).

2. **Service wrapper with injected baseUrl** — the generated `client.gen.ts`
   baseUrl (`localhost:<port>`) is a dev default only. At runtime the real URL
   is injected. `mini-document-search-client.ts` builds its own client:

   ```ts
   constructor(private readonly baseUrl: string) {
     this.client = createClient(createConfig({ baseUrl: this.baseUrl }));
   }
   // ...then per call:
   const response = await indexDocumentIndexPost({ client: this.client, body: {...} });
   ```

   The `baseUrl` comes from the DI layer (`src/di/infrastructure-provider.ts`),
   which reads env vars `NOGAI_BACKEND_MINI_*_BASE_URL` (set in `compose.yaml`,
   e.g. `NOGAI_BACKEND_MINI_KNOWLEDGE_BASE_URL: http://mini_knowledge:8007`).
   So generated code carries a throwaway localhost baseUrl; production baseUrl
   is always passed explicitly via `{ client }` on each operation call.

## Regeneration workflow (observed + inferred)

1. Start the target Python service locally (observed: configs point at
   `localhost:<port>`; **inferred**: via `docker compose` or `uvicorn`).
2. Run `pnpm gen:mini_<service>` from `frontend/nogai/`.
3. Commit the regenerated `src/sdk/<service>/` tree.

## Drift prevention

**None automated for the OpenAPI→SDK bridge (observed).** `.github/workflows/
ci.yml` has no `gen:mini_*`, `openapi`, or SDK-diff job. Because generation
requires a live service, it is not reproducible in a stateless CI checkout the
way the schema→python bridge is. The generated SDK is committed and treated as
source; drift (a backend contract change not re-generated) is caught only by
downstream TypeScript type errors, not by a regenerate-and-diff gate.
*(Inferred from absence of any CI/pre-commit reference.)*

## Other frontend↔backend contract mechanisms (brief)

- **GraphQL** (`src/graphql/server.ts`): a `graphql-yoga` server whose schema is
  auto-built from the Drizzle DB via `buildSchema(db)` (drizzle-graphql), served
  on `:4000`. This is a DB-reflection admin/dev surface, **not** a
  frontend↔Python-service contract. Not a primary contract.
- **Event schemas** (`outbox_processor`): shares `@legal-ai/event-sourcing-core`
  (a workspace TS package) — an intra-TypeScript contract, not cross-language.
